# Просмотр дневника

Показать съеденное по дням в читаемом виде. Никакого анализа, оценок и
рекомендаций — это режим чтения журнала. Свежие дни сверху.

## Шаг 1. Определи период

Из запроса: «сегодня», «неделя», «месяц», «июнь», «весь дневник», конкретные даты.
Если период не указан — покажи **весь дневник** (он отсортирован, свежее сверху —
листать удобно).

Переведи нижнюю границу периода в дату `YYYY-MM-DD`:
- «неделя» → 7 дней назад от сегодня; «месяц» → 30 дней; «июнь» → `2026-06-01`.
- «весь дневник» / период не указан → без границы.

## Шаг 2. Сгенерируй файл рендерером (без API)

Запусти готовый скрипт — он сам читает `data/diary.jsonl`, вес и норму из профиля,
собирает Markdown и кладёт его в `data/summary/дневник.md`, а также копирует в
облачный инбокс (если папка есть), чтобы читать с телефона:

```bash
scripts/render-diary.sh [YYYY-MM-DD]   # дата — нижняя граница периода; без неё — весь дневник
```

Пример за неделю: `scripts/render-diary.sh 2026-06-20`.

## Шаг 3. Покажи результат

Прочитай `data/summary/дневник.md` и выведи его пользователю как есть (формат уже
готов к чтению). Если нужен только конкретный день — покажи соответствующий блок.

Скажи, что полная версия лежит в `data/summary/дневник.md` и (если скопировалась)
в облачном инбоксе под именем `дневник.md` — можно открыть на телефоне.

## Грейсфул

- Нет `jq` → скрипт тихо выходит; в этом случае собери дневник сам, читая
  `data/diary.jsonl` по тому же формату (день-заголовок → приёмы → итог).
- Пустой дневник → скрипт пишет понятную заглушку, не падает. Предложи записать
  приём пищи.
- Нет `profile.md` → рендер просто не покажет норму и прогресс-бар, остальное в силе.

## Формат вывода (как выглядит)

```
## 23 июня — 1180 / 1600 ккал  ▓▓▓▓░  ·  ⚖️ 68.5 кг
_Б67 Ж36 У130 г_
🍳 Завтрак — овсянка, банан · 450 ккал
🍽️ Обед — гречка, котлета · 420 ккал _(±)_
🥗 Ужин — салат, рыба · 310 ккал
```

`_(±)_` — приём с низкой уверенностью оценки (`confidence: low`).
